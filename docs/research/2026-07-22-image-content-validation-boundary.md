# 微信图片模式文章是否属于首轮公共采用验证边界

> 研究日期：2026-07-22
>
> 对应决策票：[判断图片模式文章是否属于首轮核心任务边界](https://github.com/Albert-Lsk/wechat-notebank/issues/19)
>
> 范围：研究与规划，不包含产品实现

## 结论

图片模式文章应进入首轮公共采用验证的**内容边界**，但不应在缺少真人证据时直接承诺为 v0.3.0 功能。

原因是：该格式仍然承载同一个“把值得保存的公众号文章归档到本地知识库”的核心任务；现有真实样例可以正常打开，却把正文载荷放在图片列表数据中，而当前解析器只读取 `#js_content` 的 HTML。由此产生的风险不只是“不支持一种附加格式”，还可能是命令成功但 L1 没有保存正文图片。对归档工具而言，这属于必须在首次保存环节识别的内容完整性边界。

首轮合成预演必须加入一个图片模式样例，验证系统至少能做到“完整保存”或“明确拒绝”，不得把只含标题、账号和日期的文件报告为保存成功。真人验证仍允许用户自由选择文章；只有真人采用证据反复触发该格式，或地图随后认定完整性保护属于公共发布底线，才把具体支持能力写入 v0.3.0 规格。

## 已验证事实

### 微信图片模式样例的实际结构

- Issue 提供的[微信原始页面](https://mp.weixin.qq.com/s/vLedKgVI7KaDQJeF0khbDQ)在 2026-07-22 可通过真实 Chrome 正常打开；页面没有出现验证码、访问频繁、删除或停止访问提示。这排除了“样例本身已失效”作为本次结构判断的主要解释。
- 页面存在 `#js_content`，其唯一直接子节点是 `#js_image_content.image_content`。该节点包含标题、账号/地区和发布时间等页面元数据，但自身没有正文 `<img>` 节点。
- 页面运行时的 `picture_page_info_list` 包含 2 个有序图片记录，尺寸分别为 1280×1177 和 1280×1302，`cdn_url` 均指向微信第一方媒体域名 `mmbiz.qpic.cn`。页面同时加载微信自己的 `common_share_image_content` 脚本，HTML 中含有 `image_content` 标记。
- 因此，这个样例的正文载荷不是普通图文文章那种直接位于 `#js_content img` 的节点集合；图片序列由页面运行时数据描述。以上检查只记录结构、数量、尺寸和来源域名，没有复制文章正文或图片内容。

### 当前产品契约与实现

- README 把产品定义为把公众号文章内容保存为本地 Markdown，并把“链接失效或难以找回前留下一份本地副本”列为首要场景。[README 定义](https://github.com/Albert-Lsk/wechat-notebank/blob/e5a9ab9b89adcbb459b2e7f641e94f2d1896a2f2/README.md#L1-L36)
- v0.2.0 正式规格把 L1 定义为“保存微信公众号原文”，并要求单篇保存返回成功文件；完整 Agent 验收也从“保存”继续进入加工与审批。[正式规格](https://github.com/Albert-Lsk/wechat-notebank/blob/e5a9ab9b89adcbb459b2e7f641e94f2d1896a2f2/docs/superpowers/specs/2026-07-15-agent-first-knowledge-processing-spec.md#L115-L130)
- 当前抓取层只等待 `#js_content` 出现，然后返回页面 HTML。[抓取代码](https://github.com/Albert-Lsk/wechat-notebank/blob/e5a9ab9b89adcbb459b2e7f641e94f2d1896a2f2/src/lib/parser.ts#L63-L89)
- 当前解析层依次从 `#activity-name`、首个 `h1` 或 `.article-title` 读取标题，从 `#js_content`、`.article-content` 或 `article` 读取内容；清理逻辑只处理内容 HTML 中现存的 `img[data-src]`，没有读取 `picture_page_info_list`。[解析代码](https://github.com/Albert-Lsk/wechat-notebank/blob/e5a9ab9b89adcbb459b2e7f641e94f2d1896a2f2/src/lib/parser.ts#L124-L212)
- 在已验证样例中，首个 `h1` 和非空 `#js_content` 都存在，但 `#js_content` 内没有正文图片；所以仅以“标题非空 + content HTML 非空”判断成功，不能证明图片正文已经归档。这是从页面结构与当前源码共同得出的推论。
- 现有[非图文文章无法抓取](https://github.com/Albert-Lsk/wechat-notebank/issues/4)是一个真实问题报告，但只有一个样例，且报告没有记录版本、退出码或产物。因此它可以证明该边界已被用户碰到，不能证明发生频率或所有图片模式页面都具有完全相同的结构。

## 对共同 JTBD 的含义

本地图的共同 JTBD 是“把值得保存的公众号文章归档到本地知识库，并按需加工成可复用知识”。用户选择一篇文章时，通常不会先判断微信内部把它实现为普通图文还是图片模式。格式是平台的呈现选择，不是另一个用户任务。

所以首轮验证至少要回答两个问题：

1. 用户把图片模式链接交给 Agent 时，系统是否能准确识别实际正文载荷，而不是把元数据外壳当作原文；
2. 如果当前版本不支持该载荷，系统是否在写入前给出明确、可行动、可被 Agent 解释的失败结果。

这并不等于首轮必须验证 OCR 或图片理解。图片是原始来源，OCR 和大模型总结是派生解释；把二者混在 L1 会破坏“来源事实”和“机器加工”的既有边界。

## 候选归档语义

### 候选 A：有序图片作为 L1 原文，推荐的目标语义

- 识别 `image_content` 类型并记录内容类型。
- 保存标题、公众号、发布时间、来源 URL，以及平台给出的图片顺序。
- L1 正文按顺序嵌入全部内容图片；每张图片都能追溯原始媒体 URL。
- 若下载本地媒体，图片文件与 Markdown 一起处于用户掌控的知识库，并记录校验信息；只有全部正文图片成功后才报告保存成功。
- OCR、图片描述、L2/L3/L4 提炼均作为明确标注的派生内容，不能冒充原文。

这一语义最符合“本地副本”和 L1 原文的产品承诺，但“是否从远程嵌入升级为本地媒体下载”会同时影响普通图文文章，属于后续规格需要单独决定的存储边界。

### 候选 B：有序远程图片引用，最小兼容语义

- 从微信页面的图片列表生成有序图片引用，保留元数据和来源 URL。
- 保存前验证引用数量与页面图片列表一致；不得只保存页面外壳。
- 明确告知用户图片仍依赖微信 CDN，不把它描述为完整离线副本。

这一语义接近当前普通图文图片的处理方式，实施范围较小，但无法兑现“链接失效后仍有本地副本”的强承诺。

### 候选 C：明确标记不支持，安全下限

- 在检测到图片模式后返回稳定的“不支持内容类型”结果，不创建 L1 文件，不启动自动加工。
- Agent 向用户说明可保留原链接，但没有完成原文归档。

这一语义不能完成 JTBD，却优于静默生成缺少正文的“成功”文件；它可作为支持功能完成前的完整性保护。

## 建议的首轮验收证据

合成预演中应使用公开、仍可访问的图片模式样例，并只记录脱敏结构性证据：

- Agent 能从公开 README 和 Skill 自主完成调用，无需作者提示特殊命令；
- 结果明确标识内容类型或明确报告不支持；
- 成功时，L1 的图片数量和顺序与页面运行时图片列表一致，标题、公众号、发布时间和 source URL 可追溯；任一图片缺失时整体失败或回滚；
- 不把 OCR、Agent 图片描述或推测性文字写成 L1 原文；
- 开启 `autoProcess` 时，只有 L1 完整归档成功才创建待审核加工包；
- 不支持时，不留下元数据外壳文件，也不把失败计入“首次保存完成”。

真人验证中不要求参与者刻意寻找图片模式文章。若参与者自然选择了这种格式，记录它是否阻塞共同 JTBD、用户是否能理解失败说明，以及用户是否仍希望保存；不收集文章正文或知识库内容。

## 尚未验证的事实

- 图片模式在目标用户真实收藏中的出现频率，以及是否会被至少两位真人自然触发。
- 不同图片模式页面是否都使用 `picture_page_info_list`，以及单图、多图、带描述、直播照片、小程序/商品热点等变体的兼容边界。
- 微信 CDN URL 的有效期和访问控制是否允许把远程引用视为可靠归档。
- 下载媒体到本地时的文件命名、去重、失败恢复、空间占用和版权提示策略。
- 图片内容进入可选加工时，当前 Agent 是否能稳定读取本地/远程图片，以及如何把 OCR 或视觉理解与原文严格区分。

## 对路线地图的建议

把图片模式加入“定义公共采用验证协议”的合成预演用例和内容完整性停止条件；不要因为本研究单独把“完整支持图片模式”预先写进 v0.3.0。

如果合成预演确认当前版本会静默成功但丢失图片，先创建地图外维护 ticket，最低修复目标是候选 C；若真人采用证据反复支持这一需求，再在“从采用证据确定 v0.3 范围”中决定候选 A 或 B。这样既保护首轮数据完整性，也保持“证据先于功能”的地图规则。
